# Plan: Pinleaf MVP - notas post-it para desktop

## Spec Link

- Source spec: `docs/.specs/001-desktop-sticky-notes/spec.md`

## Architecture Overview

Pinleaf will be a local-first desktop application written in Python with GTK 4 and libadwaita through PyGObject. The MVP should keep product behavior small while separating UI, note state, persistence and runtime setup enough to support future distribution formats.

The application will use one process with:
- an `Adw.Application` entrypoint;
- a main window for note listing and note management;
- independent note windows for editing;
- a status/tray integration service for quick actions;
- a SQLite repository for durable local state;
- a small application service layer that coordinates notes, autosave and windows.

The initial runtime target is direct execution from the development environment. Packaging-specific assumptions must stay out of the application core.

## Proposed Module Boundaries

The implementation should start with a package layout similar to:

```text
pinleaf/
  __main__.py
  app.py
  config.py
  models.py
  services/
    note_service.py
    autosave.py
  storage/
    sqlite_store.py
    migrations.py
  ui/
    main_window.py
    note_window.py
    dialogs.py
    tray.py
  resources/
    styles.css
tests/
```

Responsibilities:
- `app.py`: creates the GTK/libadwaita application, wires dependencies and owns application lifecycle.
- `config.py`: resolves XDG-compatible data paths and runtime settings.
- `models.py`: defines note data structures and supported color values.
- `storage/sqlite_store.py`: reads and writes notes using SQLite.
- `storage/migrations.py`: creates and upgrades the local schema.
- `services/note_service.py`: owns note operations such as create, update, delete, list and restore state.
- `services/autosave.py`: debounces frequent edits before writing to SQLite.
- `ui/main_window.py`: lists notes and exposes create, open, delete and color-aware previews.
- `ui/note_window.py`: renders one editable note window and forwards edits/state changes.
- `ui/dialogs.py`: confirmation and error dialogs.
- `ui/tray.py`: status/tray icon actions for create note, show/focus main window and quit.

## Data Model

SQLite is the MVP persistence mechanism. The database should live under the user data directory, for example:

```text
~/.local/share/pinleaf/pinleaf.sqlite3
```

Initial schema:

```sql
CREATE TABLE schema_version (
  version INTEGER NOT NULL
);

CREATE TABLE notes (
  id TEXT PRIMARY KEY,
  content TEXT NOT NULL DEFAULT '',
  color TEXT NOT NULL,
  width INTEGER NOT NULL,
  height INTEGER NOT NULL,
  position_x INTEGER,
  position_y INTEGER,
  is_open INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  deleted_at TEXT
);

CREATE INDEX idx_notes_deleted_updated
  ON notes (deleted_at, updated_at);
```

Notes:
- `id` should be a UUID string generated by the application.
- `color` should be constrained in application code to the supported MVP colors: `yellow`, `green`, `blue`, `pink`.
- `deleted_at` enables soft delete for safer local recovery during development; deleted notes are excluded from normal reads.
- `position_x` and `position_y` are nullable because some desktop environments do not support reliable window position restoration.
- Timestamps should use ISO 8601 UTC strings.
- Schema versioning should be explicit from the first migration.

## UI Approach

Use libadwaita for the application shell and GTK widgets for note editing.

Main window:
- shows an empty state with a create action when no notes exist;
- lists non-deleted notes with preview text, color indicator and last update time;
- opens or focuses a note window when a note is activated;
- exposes delete with confirmation for non-empty notes.

Note window:
- opens directly into editable text;
- uses a visual style per note color through CSS classes or style providers;
- saves text changes through debounced autosave;
- saves size changes after resize events or close;
- closes the window without deleting the note;
- supports color selection with the four MVP colors.

Status/tray:
- exposes create note, show/focus main window and quit;
- should be implemented behind a small adapter because tray/status support varies across Linux desktop environments;
- if the target desktop cannot display a status icon, the application should remain functional through the main window.

## Window State

Each note window should be tracked by note id. The application service should prevent duplicate windows for the same note by focusing an existing note window when possible.

On startup:
- load non-deleted notes from SQLite;
- restore windows marked as open;
- apply saved width and height;
- attempt position restore only when technically supported;
- keep running if position restore is ignored by the window manager.

On note window close:
- persist `is_open = 0`;
- keep the note listed in the main window.

On app quit:
- persist current content and window dimensions for open note windows before shutdown.

## Autosave

Autosave should debounce content updates to avoid writing on every keystroke. A practical first target is 300-750 ms after the latest edit.

Required behavior:
- content changes eventually write to SQLite without a manual save button;
- close and quit paths flush pending autosave work;
- save failures show a concise error message and are logged to stderr or local logs;
- failed saves should not mark the in-memory content as safely persisted.

## Error Handling

Startup database errors:
- do not overwrite an unreadable or invalid database;
- show a clear fatal error dialog if notes cannot be loaded safely;
- log the exception details for diagnosis.

Write errors:
- keep the UI responsive where possible;
- show that the latest change may not have been saved;
- log details for development diagnosis.

Partial read recovery:
- if a future schema or row-level corruption allows reading valid notes independently, list valid notes and report skipped failures;
- for the initial simple schema, database-open or migration failure may be treated as fatal to avoid accidental data loss.

## Security and Privacy

- Do not execute note content.
- Do not auto-open URLs, files or shell commands detected in note text.
- Create the data directory and database with user-only permissions where the platform allows it.
- Do not add telemetry or network calls.
- Keep all note content local to the SQLite database.

## Accessibility

- Main actions must be reachable by keyboard.
- Buttons and color controls should have accessible labels.
- Color must not be the only way to identify a note; the list also shows preview text and updated time.
- The note editor should use standard GTK text widgets to inherit platform accessibility behavior.

## Performance Targets

The MVP should satisfy the spec targets through simple design:
- load and list up to 100 short notes during startup;
- keep note editing responsive for up to 10,000 characters;
- avoid blocking the GTK main loop with unnecessary synchronous write bursts.

SQLite access can remain synchronous for the MVP if writes are small and debounced. If UI stalls appear during verification, move storage operations behind an idle callback or worker boundary.

## Rollout and Distribution

Development execution:
- support direct execution from the repository, preferably through `python -m pinleaf` or an equivalent project script.

Future distribution readiness:
- isolate data path resolution in `config.py`;
- avoid hard-coded repository-relative resource paths in UI code;
- load CSS/resources through a single resource helper that can later adapt to Flatpak or package installation;
- keep app id, desktop metadata and icon naming centralized.

Packaging work itself is out of scope for the MVP implementation unless a later spec adds it.

## Testing and Verification

Automated tests should focus first on non-GUI behavior:
- SQLite migration creates the expected schema.
- Creating a note persists required fields.
- Editing content updates `content` and `updated_at`.
- Deleting a note hides it from normal list queries.
- Color validation accepts only supported colors.
- Autosave flush path writes pending content.

Manual verification should cover GTK behavior:
- launch empty app and create first note;
- edit, close, reopen and restart with persisted content;
- delete with confirmation and cancel flow;
- change note color and restart;
- use status/tray actions to create note, show main window and quit;
- verify app remains usable when position restore is unavailable.

## Alternatives Considered

Structured file storage:
- rejected because the spec now requires SQLite and future migrations are easier with schema versioning.

Single-window note editor:
- rejected for the MVP because the spec requires independent post-it-style note windows.

No status/tray integration:
- rejected because the resolved scope includes quick access through a status/tray icon.

Immediate packaging work:
- deferred because the MVP supports direct development execution first, while preserving architecture for later distribution.

## ADR Impact

No ADR is required before implementation if the team treats Python, GTK/libadwaita, SQLite and status/tray support as accepted MVP constraints from the spec.

Create an ADR later if the project commits to a specific distribution model, changes the storage engine, or introduces a background daemon or cross-device sync.

## Implementation Risks

- Tray/status support differs across Linux desktop environments; keep it behind an adapter and preserve full functionality through the main window.
- Window position restoration may be unreliable or unsupported under Wayland; treat saved position as best-effort.
- Synchronous SQLite writes can cause UI stalls if autosave is too aggressive; debounce writes and flush intentionally on close/quit.
- Corrupt database recovery is limited in the MVP; avoid automatic overwrite and log enough context for manual diagnosis.
